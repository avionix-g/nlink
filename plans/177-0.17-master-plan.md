# Plan 177 — 0.17 cycle master

The 0.16 cycle shipped (`v0.16.0` tagged + both crates published
2026-05-25). The work itself was substantial; the *process* of
shipping it surfaced a coherent set of issues — most around
defensive coding patterns the lib should have but doesn't. This
plan catalogues them, ranks by user-visible severity, and
orchestrates the per-issue plans (170-176) that fix them.

## 1. Theme of the cycle

**"The hidden hang."** The integration-test CI run that gated
0.16's cut hung for 22 minutes on its first execution. Step B
+ C of the cut activation (Plan 167) localized the cause to a
single `recv_msg().await?` loop in `Connection::<Nftables>::send_batch`
that lacks both a sequence-number filter and a timeout. We shipped
0.16.0 with the 7 affected integration tests `#[ignore]`'d and
Plan 170 queued.

But the *pattern* — recv-loops that can hang forever if the
kernel response shape doesn't match expectations — is wider than
one function. **Plan 172** audits every recv-loop in the lib;
**Plan 171** introduces a default operation timeout so future
"hidden hangs" surface as `Error::Timeout` instead of silent
indefinite blocks; **Plan 174** wires `tracing-subscriber` into
the integration test harness so the next "hidden hang" comes
with debug logs.

Two unrelated UX papercuts also surfaced and rate their own
plans (**Plan 173**: parse-error `From`-impls; **Plan 175**:
release-cut tooling).

## 3. Sequencing rationale

1. **170 first** — un-ignores the 7 reconcile tests, restores
   the regression net for the rest of the cycle's work.
2. **171 next** — default timeout is independent of 170 but
   benefits from 170's lib changes being settled (touches the
   same socket layer). Without 171, every future Plan-170-class
   bug recurs as a silent hang.
3. **172 after 170 + 171** — the audit's recommended fix for
   each problem loop is "switch to the seq-filtered shape from
   170 and wrap in the default timeout from 171." So 170 + 171
   are prerequisites for 172's per-loop edits.
4. **173** — independent additive fix; can land anytime.
   Sequenced after the recv-loop work to keep the lib changes
   bundled.
5. **174** — would have made 170 easier to diagnose (debug
   logs from tracing-subscriber would have surfaced the hang
   point in the first CI run instead of the 3 push-watch-fix
   iterations). Land mid-cycle so 0.18's debug story is better.
6. **175** — release tooling. Apply to the 0.17 cut.
7. **176** — strategy doc; doesn't unblock anything; land at
   maintainer's leisure.
8. **169 Phase 3** — the design call deferred from 0.16. Land
   last so any related diagnostics work folds into the formula.

## 4. Themes that did NOT surface as 0.17 work

Out of scope for this plan:

- **Plan 152** (aya / Prometheus / OTel integration showcases) —
  remains deprioritized per the 0.16 cycle decision. No new
  signal indicates it should bubble up.
- **Public-API stability work** beyond what Plan 163 already
  did — `cargo-semver-checks` is enforced in CI, so semver
  drift is structurally blocked.
- **MSRV bump** — 0.16 raised MSRV to 1.95. No need to revisit
  until a 1.95 feature gains value the lib wants.
- **New kernel families** beyond DPLL / net_shaper — neither
  the macro stack nor any user request currently points at a
  next family.

## 6. Process improvements baked in this cycle

Compared to 0.16:

- **CI surfaces issues earlier** — Plan 174's tracing-subscriber
  init means the next "hidden hang" comes with debug logs in CI
  immediately, no need for the 3-push iteration loop that
  diagnosed Plan 170 (Plan 167 §B + §C).
- **Release cut is scripted** — Plan 175's
  `scripts/cut-release.sh` makes the Plan 167 runbook
  executable, with checks for the dry-run-inversion edge case
  and the GitHub release body length.
- **Recv-loop hang class is closed** — Plans 170 + 171 + 172
  remove the pattern that caused Plan 170; the default timeout
  is the safety net that catches anything missed.

## 7. Risks

- **Plan 170 fix doesn't reproduce locally**: same risk that
  blocked the original cut. Mitigation: iterate via CI pushes
  on the 0.17 branch's draft PR; the fix is small and easy to
  iterate on. Plan 170 §7 documents this.
- **Plan 171 default-timeout choice**: too short causes flakes
  on slow kernels / slow CI; too long doesn't help users
  diagnose. Plan 171 §4 proposes 30s with a knob to override.
- **Plan 172 audit surfaces more bugs than 170**: if the audit
  finds 3+ additional Plan-170-class bugs, the cycle's scope
  grows. Mitigation: triage each to either fix-in-place
  (cheap) or `#[ignore]` + follow-up plan (recover).

## 8. Out-of-scope follow-ups (post-0.17)

- **0.18 release tooling polish** — once 0.17 cut runs through
  `scripts/cut-release.sh`, expect minor refinements.
- **Per-protocol connection pools** (`ConnectionPool<Nftables>`,
  `<Generic>`, etc.) — Plan 159's generic pool already works
  via `connection_for_async`; the typed-per-protocol convenience
  variants are nice-to-have.
- **kernel-version matrix CI** — currently bookworm-only. A
  multi-version matrix (Ubuntu 22/24 + Fedora 40/41) would
  have caught Plan 170 earlier. Cost vs. benefit unclear.

End of plan.
